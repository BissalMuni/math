**LLM 기초**:
- 토큰화, 임베딩, 어텐션, 트랜스포머 레이어
- 파라미터/가중치, Softmax 확률 계산, 온도(temperature), 컨텍스트 윈도우
- **핵심 인사이트**: 파라미터 = 분산된 패턴 인코딩 (룩업 테이블 아님)

**법률 AI 설계 결정**:
- 낮은 temperature만으로는 환각 방지 불충분 → RAG 필요
- 참조 조문을 함께 제공하면 LLM 응답 개선 (Graph RAG 방향과 일치)
- **Neo4j Community Edition** (GPL 오픈소스) 선택
- 대상 도메인: 취득세, 재산세, 양도소득세, 종부세
- 1단계 법령 수집 완료

**핵심 아키텍처 결정**:
- 양도소득세는 취득 시점 법령이 적용 → 조문에 시행일/종료일을 노드 속성으로 저장하는 시간적 버전 관리 필수

## 6. 500b 모델을 70b으로 양자화하기

- **날짜**: 2026-03-18
- **URL**: https://claude.ai/chat/987f4fd9-afb1-4c88-ae06-582b53becd4b
- **주제**: 모델 압축 기법

### 요약

500B → 70B 파라미터 축소가 BMP → JPG 압축과 같은지 질문.

**압축 기법 3종 비교**:

| 기법 | 비유 | 설명 |
|------|------|------|
| 양자화 | BMP→JPG | 같은 파라미터, 낮은 정밀도 (FP32→INT4) |
| 지식 증류 | 4K→720p 리사이즈 | 교사 모델 → 학생 모델로 지식 이전 |
| 프루닝 | 저중요 픽셀 삭제 | 중요도 낮은 가중치 제거 |

**결론**: BMP→JPG는 양자화에 더 정확한 비유. 500B→70B는 지식 증류. DeepSeek-R1이 실제 예시.

## 8. LLM의 어텐션 메커니즘과 파라미터 가중치

- **날짜**: 2026-03-19
- **URL**: https://claude.ai/chat/0f4081ee-6668-4452-9979-924f1f895905
- **주제**: 트랜스포머 전체 파이프라인 + 하드웨어

### 요약

트랜스포머 전체 파이프라인을 처음부터 끝까지 심층 탐구.

**커버된 주제**:
- BPE 토큰화, 임베딩 테이블, 위치 인코딩
- 어텐션 메커니즘: Q/K/V 행렬
- 피드포워드 레이어, Softmax, 토큰 선택
- Wq, Wk, Wv가 역전파로 학습되는 방식

- 어휘 사전과 해시맵
- 다국어 토큰화 효율 차이
- GPU 메모리 요구사항
- 트랜스포머 논문 역사 ("Attention is All You Need", 2017, 8명 저자)

**핵심 인사이트**:
- 5토큰 → 25회 어텐션 계산 (n²) 독립 도출
- RNN: 줄 단위 읽기 vs 어텐션: 페이지 전체 속독 (본인이 만든 비유)
- Google이 트랜스포머 논문 출판을 후회했다는 Claude의 미검증 주장을 직접 지적하고 수정 요구

## 11. Ontology 지식그래프와 LLM의 활용

- **날짜**: 2026-03-20
- **URL**: https://claude.ai/chat/b1b15871-50dc-4de5-b344-0fccdb70cdf1
- **주제**: 법률 지식그래프 아키텍처 설계

### 요약

한국 부동산 세금 Graph RAG 시스템의 지식그래프 구조를 상세 설계한 핵심 대화.

**세금 도메인**:
- 국세: 양도소득세, 종부세 (기재부, 국세청)
- 지방세: 취득세, 재산세 (행안부, 지자체)

**해석 권위 계층**:
```
헌법재판소
  ↓
대법원 판례
  ↓
하급심 판결
  ↓
조세심판원 / 지방세심판원
  ↓
장관 해석 지침
  ↓
법령 조문
```

**노드 설계**:
- 법령, 조문, 버전화된 조문 텍스트, 판결, 판례, 해석 지침, 지자체

**관계 설계** (모두 속성 포함):
- 준용, 위임, 예외적용, 하위법령, 버전, 해석, 쟁점조문, 파기환송

**핵심 인사이트**:
- 노드와 관계 모두 자체 속성을 가짐
- Neo4j의 관계 속성 인라인 저장 → 조인 테이블 불필요 → LLM 컨텍스트 전달 부담 감소
